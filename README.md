# 📱 离线二维码文件传输系统 V2.0

一个纯浏览器实现的、完全离线的文件传输工具，通过二维码序列在没有网络连接的环境下传输文件。**不需要任何服务器，不需要安装软件，仅需浏览器和摄像头。**

## ✨ 特性

- 🌐 **纯浏览器实现** - 无需安装，无需后端，打开即用
- 📶 **完全离线工作** - 不依赖任何网络连接，适合涉密环境、无信号区域
- 🔒 **数据安全** - CRC32 校验确保每个分片完整性，文件指纹防止串扰
- 📦 **高效压缩** - 使用 pako (zlib) 最高压缩级别，减少二维码数量
- 🎯 **Base45 编码** - 专为二维码优化的编码方案，膨胀率仅 22%（比 Base64 节省约 11% 空间）
- 📁 **支持任意文件** - 文本、图片、PDF、压缩包等
- 🇨🇳 **中文友好** - 完美支持中文文件名（Base64 编码传输）
- 💾 **断点续传** - 接收端自动保存进度，关闭浏览器或中断后仍可继续
- ⌨️ **便捷导航** - 发送端支持键盘（←↑/→↓）和鼠标滚轮切换二维码
- 🎨 **现代化界面** - 左右分栏设计，二维码单视图显示，操作直观

### 3. 使用步骤

**发送端（电脑）：**
1. 用浏览器打开 `send.html`
2. 点击「选择文件」选取要传输的文件
3. 调整分片大小（默认 1200 字符，可增可减）
4. 点击「生成二维码」
5. 使用键盘方向键或鼠标滚轮切换二维码，按顺序展示给接收端扫描

**接收端（手机/另一台电脑）：**
1. 用浏览器打开 `receiver.html`
2. 点击「开始扫描」并授权摄像头权限
3. 对准发送端屏幕上的二维码，按顺序扫描所有数据分片（最后扫描文件名分片）
4. 扫描完成后自动重组文件，点击「下载文件」保存

## 🔧 技术细节

### 整体流程

```
发送端：
文件 → 读取为 ArrayBuffer → pako.deflate (level 9) 压缩 
    → Base45 编码整个压缩数据 → 按字符数分片 
    → 每个分片包装成 JSON {i, t, h, f, d} 
    → 生成二维码图片 → 显示并支持导航

接收端：
扫描二维码 → 解析 JSON → 校验 CRC32 → 存储分片（IndexedDB）
    → 检测是否收齐所有数据分片和文件名分片 
    → 自动拼接 Base45 字符串 → Base45 解码 
    → pako.inflate 解压 → 创建 Blob 下载
```

### 分片数据结构

**数据分片** (`t` 字段不存在或为 `'data'`)：
```json
{
  "i": 0,            // 分片索引（从0开始）
  "t": 5,            // 总分片数
  "h": "abc12",      // CRC32 校验码（5位36进制）
  "f": "x9k2m",      // 文件指纹（5位随机字符串，标识同一文件）
  "d": "ABCD..."     // 分片数据（Base45 编码的片段）
}
```

**文件名分片** (`"t": "fn"`)：
```json
{
  "t": "fn",         // 类型标识
  "f": "x9k2m",      // 文件指纹
  "n": "JUU5JTlG...",// Base64 编码的文件名（支持中文）
  "s": 102400,       // 原始文件大小（字节）
  "ts": 1712345678,  // 时间戳
  "tc": 5,           // 数据分片总数
  "h": "def34"       // CRC32 校验（对除 h 外的字段计算）
}
```

### 为什么选择 Base45？

- **二维码容量限制**：一个二维码最多存储约 2953 字节（Version 40, L 纠错）。
- **Base64 膨胀率 33%**：每 3 字节变成 4 字符，有效载荷约 2200 字节。
- **Base45 膨胀率 22%**：每 2 字节变成 3 字符，有效载荷约 2420 字节，**每分片多传输约 10% 数据**，大幅减少分片数量。

项目中 Base45 采用纯 JavaScript 实现，符合 RFC 9285 标准。

### CRC32 校验

- **发送端**：对每个分片的 `d` 字段（Base45 字符串）计算 CRC32，取 36 进制后 5 位作为 `h`。
- **接收端**：收到分片后重新计算 `d` 的 CRC32，与 `h` 比对，不一致则丢弃该分片。
- **文件名分片**：对 `t, f, n, s, ts, tc` 字段的 JSON 字符串计算 CRC32。

使用 `pako.crc32` 保证与 pako 库一致，备用实现兼容无 pako 环境。

### 发送端导航与交互

- **左右分栏布局**：左侧固定宽度 400px 设置参数，右侧自适应显示当前二维码。
- **键盘事件**：`←` / `↑` 上一张，`→` / `↓` 下一张（全局监听，页面任意位置生效）。
- **鼠标滚轮**：滚动切换二维码，带 300ms 防抖，避免快速滚动触发多次。
- **边界限制**：不循环，到达首张或末张时按钮禁用，键盘/滚轮无效。
- **二维码尺寸**：实时调整显示尺寸（200~800px），已生成图片会同步缩放。
- **分片大小**：500~1500 字符，推荐 1200。可根据文件大小和二维码容量动态调整。

### 接收端自动重组与存储

- **IndexedDB 持久化**：使用 `localforage` 库存储已接收的分片和文件信息，刷新页面或关闭浏览器后数据不丢失。
- **自动检测完成**：每次收到新分片后检查 `receivedDataChunks.size >= totalDataChunks` 且 `fileNameData` 存在，满足条件立即调用 `assembleFileAutomatically()`。
- **重组流程**：
  1. 按索引顺序拼接所有数据分片的 `d` 字段 → 完整 Base45 字符串。
  2. `base45.decode()` 得到压缩后的 Uint8Array。
  3. `pako.inflate()` 解压得到原始文件字节。
  4. 创建 Blob 并生成下载链接，显示下载按钮。
- **容错**：手动输入备用（调试用），清除数据按钮重置所有存储。

### 压缩优化

```javascript
const compressed = pako.deflate(new Uint8Array(fileBuffer), { level: 9 });
```
- `level: 9` 为最高压缩级别，对文本类文件（如 JSON、HTML、日志）可压缩至原大小的 20%~40%，对已压缩文件（图片、视频）效果有限但无负面影响。

## 📂 代码结构说明

### send.html 核心函数

- `generateShortFileId()`: 生成5位随机文件指纹（时间戳+随机数转36进制）。
- `encodeFileName()`: 将原始文件名 UTF-8 编码后 Base64，保证中文传输。
- `processFileChunks()`: 压缩 → Base45 整体编码 → 按字符分片 → 构造分片对象。
- `generateSingleQR()`: 异步生成单个二维码图片，支持深蓝色文件名二维码。
- `showQRAtIndex()` / `goPrev()` / `goNext()`: 单视图导航逻辑。
- 键盘/滚轮事件绑定 + 防抖。

### receiver.html 核心函数

- `processChunkData()`: 解析 JSON，区分数据/文件名分片，校验 CRC32，存储到 IndexedDB。
- `checkCompletion()`: 检查所有分片是否收齐，触发自动重组。
- `assembleFileAutomatically()`: 拼接 Base45 → 解码 → 解压 → 创建下载。
- `loadSavedChunks()`: 页面加载时从 IndexedDB 恢复进度。
- `updateUI()`: 更新进度条、分片网格显示。

## ❓ 常见问题

**Q: 扫描时提示“找不到摄像头”？**  
A: 请确保已在浏览器权限设置中允许摄像头访问，且设备有摄像头硬件。部分浏览器需要 HTTPS 环境（本地 file:// 协议也可能工作，但推荐使用本地 HTTP 服务器）。

**Q: 为什么需要 Base45？直接用 Base64 不行吗？**  
A: Base64 膨胀率更高，同样大小的二维码可容纳的有效数据更少，分片数量更多。Base45 是专为二维码设计的标准，能提高约 10% 的传输效率。

**Q: 分片大小设置为多少合适？**  
A: 推荐 1200 字符。过小会导致二维码数量增多，过大会超过 JSON 长度限制（≥1800 字符可能无法生成二维码）。若生成时提示 JSON 过大，请减小分片大小。

**Q: 接收端能兼容旧版本（Base64 编码）的二维码吗？**  
A: 不兼容。此版本使用 Base45 编码，与之前的 Base64 版本数据格式不同。如果之前有旧二维码，请用旧版接收端。

**Q: 为什么接收端没有“重组文件”按钮？**  
A: 为了提高自动化程度，接收端检测到所有分片完成后会立即自动重组，无需手动点击。重组成功后显示下载按钮。
